# Union Arena 線上對戰平台 MVP — 討論整理

---

## 1. 產品目標（Product Goal）

打造一個讓**沒有固定牌友**的實體 TCG 玩家，也能像線上遊戲一樣：

- 登入平台
- 找到陌生玩家
- 建立遠端對戰
- 透過視訊進行實體卡牌比賽
- 雙方自行操作計分

> 初期目標不是取代遊戲本身，而是解決：「想玩 Union Arena，但沒有朋友或固定牌友時，如何快速找到對手並完成一場可信任的遠端比賽。」

---

## 2. MVP 範圍（第一階段）

### 必做功能

#### A. 帳號系統

- 玩家可以：
    - 註冊 / 登入
    - 建立玩家身份
    - 查看自己的戰績
- 初期建議：
    - Google Login
    - Discord Login
- 避免自行維護密碼系統。

---

<!-- 移除Ranking系統，B為Match Room，原C往前一位，以下編號往前一位 -->

#### B. 比賽房（Match Room）

- 核心流程：
    1. 玩家 A 登入
    2. 建立比賽
    3. 等待玩家 B
    4. 開始 Match
    5. 進入房間

- Match 狀態：
    - `WAITING`
    - `READY`
    - `PLAYING`
    - `FINISHED`

---

#### C. 視訊串流

- 目的：讓雙方看到對方牌桌。
- 初期：採用 WebRTC。
- 架構：
    - 手機（Camera 裝置） → WebRTC → 對手觀看
    - 手機僅需固定拍攝牌桌，操作簡單
- 建議：
    - 使用 QR Code 綁定 Camera
    - 流程：
        1. 電腦登入
        2. 建立 Match
        3. 產生 QR Code
        4. 手機掃描
        5. 手機變為攝影機

---

#### D. 計分盤

- 雙方玩家自行操作
- 原則：「信任制」
- 不做：
    - 自動判定勝負
    - AI 判牌
    - 裁判系統
- 流程：
    1. 玩家操作加分（+1）
    2. Backend 驗證
    3. 分數同步給雙方

---

#### E. 比賽結果

- 比賽結束：
    - 玩家選擇 Winner（Player A 或 Player B）
- 更新：
    - Match Result

---

## 3. MVP 不包含功能

**目前刻意排除：**

- 不做 Ranking／排名 與 ELO 系統

- 不做自動裁判
    - 例：牌面辨識、違規偵測、AI 判定

- 不建完整 Tournament 系統
    - 例：淘汰賽、瑞士制、賽程管理

- 不做 Replay（錄影保存）

- 不做反作弊
    - 例：多鏡頭、AI 監控、手牌辨識

- 不做 Native App
    - 第一階段皆採 Web

---

## 4. 建議技術架構

### Frontend

- Web App
    - 建議：React / Next.js
    - 支援：
        - Desktop：比賽控制、計分
        - Mobile：Camera Mode

### Backend

- MVP 不拆微服務，採 Modular Monolith
- modules:
    - auth
    - user
    - match
    - score
    - video

### Database

- 建議：PostgreSQL

---

## 5. Database 初版設計

- **users（玩家資料）**
    - id
    - username
    - email
    - avatar
    - created_at

- **matches（比賽紀錄）**
    - id
    - player1_id
    - player2_id
    - status
    - winner_id
    - started_at
    - ended_at

- **match_scores（比分）**
    - id
    - match_id
    - player_id
    - score
    - updated_at

- **match_events（事件紀錄）**
    - id
    - match_id
    - player_id
    - action
    - created_at

    > 用途：保留操作紀錄，例：Alice 在 10:32 做 SCORE_UPDATE +1，方便未來查問題。

---

## 6. 即時通訊架構

- 需要：WebSocket
- 用途：
    - 比分同步
    - Match 狀態同步
    - 房間事件
- 建議：Socket.IO
- 事件：
    - MATCH_CREATED
    - PLAYER_JOINED
    - SCORE_UPDATED
    - MATCH_FINISHED

---

## 7. WebRTC 架構

- 需要：Signaling Server
    - 負責：
        - SDP 交換
        - ICE 交換
        - STUN（協助 NAT 穿透）
    - TURN（處理無法 P2P 的環境）
- 建議：coturn
- 流程：
    ```
    Player A Phone
           |
        WebRTC
           |
    Player B Browser
    ```

## 9. 開發優先順序建議

### Phase 0 — Prototype

- 目標：驗證核心流程
- 完成項目：
    1. 登入
    2. 建立房間
    3. 兩人加入
    4. 手機 Camera 串流
    5. 看到對方
    6. 比分同步

### Phase 1 — MVP

- 加入功能：
    - Match History
    - 基本 UI
    - 使用者資料

### Phase 2 — 社群化

- 加入功能：
    - 好友
    - 戰績
    - Deck Profile
    - 聊天
    - Tournament

### Phase 3 — 專業競技化

- 加入功能：
    - 裁判
    - 錄影
    - Replay
    - 反作弊
    - 官方賽事

## 10. 目前最重要的技術風險

| 風險分級 | 項目              | 問題說明                                                                 |
|-----------|-------------------|------------------------------------------------------------------------|
| 高風險    | 1. WebRTC 穩定性   | NAT、行動網路、WiFi、TURN 成本                                           |
| 高風險    | 2. 手機Camera UX   | 掃QR即用、不要求複雜登入、不容易斷線                                     |
| 中風險    | 3. Match State 管理| 玩家離線、雙方同時提交結果、房間恢復                                     |
